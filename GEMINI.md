# 《台灣三國誌》AI 開發規範 (GEMINI.md)

> 本文件為 AI 開發者的行為準則與環境配置，所有參與本專案的 AI 模型皆須遵守。

---

## 一、AI 開發體制

本專案**全程由 AI 主導開發與企劃**，人類僅負責：
- 測試執行與結果回報
- 提出建議或疑問
- 最終審核與決策

**AI 是開發與企劃的主力**，需主動推進進度、發現問題、提出解決方案。

---

## 二、模型分工與切換指引

| 角色 | 模型 | 適用場景 |
| :--- | :--- | :--- |
| **規劃 / 架構設計** | Claude Opus 4.6 (Planning) | GDD/DDD 設計、架構決策、複雜問題分析、工作清單規劃 |
| **複雜實作** | Gemini 3.1 Pro High | 核心系統實作（網路層、Session、戰鬥引擎、AOI）、重構、跨模組整合 |
| **簡單實作** | Gemini 3 Flash | 單檔修改、UI 微調、bug fix、文件更新、測試撰寫 |

### 切換提醒規則
- 工作清單中的每個任務應標註建議使用的模型
- 遇到需要跨模組設計決策時，建議切換至 **Opus 4.6 Planning**
- 單純的 CRUD / 樣板代碼可交給 **Flash**
- 涉及併發、加密、協定實作等複雜邏輯時使用 **Pro High**

## 三、開發環境

| 項目 | 規格 |
| :--- | :--- |
| **開發機** | Mac Air (macOS) |
| **IDE** | Google Antigravity |
| **Server 執行環境** | Linux |
| **Client 目標平台** | Windows / Linux (Ubuntu) / macOS |
| **未來擴充** | iOS / Android（目前不列入優先） |

### 交叉編譯注意事項
- 伺服器二進位需可在 Linux amd64 上執行
- 客戶端需支援三平台編譯（`GOOS=windows/linux/darwin`）
- 避免使用 CGO 依賴（已選用 `modernc.org/sqlite` 免 CGO）
- Ebiten 在三平台皆有原生支援，無需額外處理

---

## 四、權限與安全

- **專案目錄全權開放**：AI 對專案目錄擁有完整讀寫權限
- **不使用 sandbox**：直接操作檔案系統
- **操作守則**：
  - ⚠️ 刪除檔案前務必確認不影響其他模組
  - ⚠️ 覆寫檔案前確認變更範圍
  - ⚠️ 修改 `proto/message.proto` 後必須重新生成 `resource/`
  - ⚠️ 不可修改 `docs/` 下已 approved 的 Spec 文件，除非經過修訂流程

---

## 五、開發流程 SOP

```
1. 確認任務（從 phase_task.md 取得）
2. 選擇適合的模型執行
3. 實作代碼
4. 自我驗證（go build / go test）
5. 更新 task.md 勾選完成項目
6. 如有架構變更 → 更新 DDD.md 修訂紀錄
7. 如有遊戲設計變更 → 更新 GDD.md 修訂紀錄
```

---

## 六、文件優先原則

- **No Premature Implementation**：在 Spec 和 Task 確認並收到「Start Implementation」指令前，**絕對不要**修改任何專案代碼
- **Document Priority**：所有變更必須先反映在相關文件中
- **Revision History**：Spec 和 Task 的任何變更必須更新「修訂紀錄」表格

---

## 七、模型交接備忘

由於可能會換模型開發，每次交接時需確保：

1. **task.md 狀態最新**：已完成/進行中/未開始 的標記正確
2. **DDD.md 反映現狀**：架構決策有紀錄
3. **代碼可編譯**：`go build ./...` 無錯誤
4. **無半成品**：不留下未完成的函數骨架（要嘛完整實作，要嘛不動）
